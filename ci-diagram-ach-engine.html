Continuum Intelligence v3 — ACH Research Engine

How equity research is generated: every LLM call, prompt, schema field, and validation rule

1. Research Generation Pipeline

Data Gathering
~30 sec
Evidence Analysis
~20 sec
Hypothesis Synthesis
~80 sec
Merge & Validate
~5 sec
Publish
~2 sec

Parallel Execution Tracks

Track 1: Data Gathering Synchronous — Block All Other Tracks
Pull price data from Yahoo Finance, ASX announcements from RSS, news from DuckDuckGo, commodity futures for macro context. ~30 seconds total.
Track 2: Evidence Update/Creation Gemini 2.0 Flash ~20 sec
Produce or update 10 evidence domain cards. Runs in parallel with Tracks 3, 4, 5, 6.
Track 3: Hypothesis Synthesis Claude Sonnet 4.6 ~80 sec
Four hypotheses (N1-N4) with narrative, verdict, and skew computation. Runs parallel with Track 2, awaits Track 1 completion.
Track 4: Structure Update Gemini 2.0 Flash ~15 sec
Discriminators, tripwires, technical analysis, gaps. Runs in parallel with Track 3.
Track 5: Price Driver Analysis Claude Sonnet 4.6 ~120 sec (optional)
Broker research, social sentiment, macro period returns. Optional, runs in background.
Track 6: Gold Overlay NotebookLM + Claude ~90 sec (gold stocks only)
Deep fundamental analysis via NotebookLM agent. Gold stocks (FMG, RMS, OBM, etc.) only.

2. Research JSON Schema — 8 Sections

Click each card to expand and see the full schema with field descriptions.

Section 1: Price History
365-day price series from Yahoo Finance
▼
  • priceHistory array Array of daily price objects: { price: float, currency: string, date: YYYY-MM-DD }
  • currentPrice float Most recent close price in A$
  • currency string ISO 4217 code (AUD)
  • date string Most recent price date (YYYY-MM-DD)
Section 2: Hero / Home Page
Summary card for home page tile
▼
  • heroDescription string One-line: "BHP is a diversified resources major with exposure to iron ore, coal, copper, and oil."
  • heroCompanyDescription string (HTML) 4-6 sentence overview with links and formatting
  • heroMetrics array 4 key metrics: [{ label: "Mkt Cap", value: "A$195B" }, ...]
  • hero.skew enum UPSIDE | DOWNSIDE | BALANCED (computed from hypothesis scores)
  • hero.skew_description string 2-3 sentence skew rationale
  • hero.embedded_thesis string What the current price assumes (3-5 sentences)
  • hero.position_in_range object { bear: 10.50, base: 12.25, bull: 15.00, catalyst: 18.75 }
  • hero.next_decision_point object { event: "Q2 earnings", date: "2026-07-15", metric: "EBITDA", thresholds: { positive: ">$5.2B", negative: "<$4.8B" } }
Section 3: Hypotheses
ACH core: 4 competing scenarios
▼

Four hypotheses (N1-N4) with probabilistic weights that sum to 100%.

Hypothesis Direction Key Fields
N1 upside title, score (%), description, supporting[], contradicting[], requires[]
N2 any Base case — usually highest weight
N3 downside Bear case with evidence
N4 any Tail risk, low probability
N1 35%
N2 40%
N3 20%
N4 5%
Section 4: Narrative
The story: what & what price assumes
▼
  • theNarrative string Dominant theme right now (3-5 sentences, updated with new data)

  • evidenceCheck string How strong is the evidence base for this narrative
  • narrativeStability enum robust | fragile | uncertain — how easily could the narrative break

Section 6: Verdict & Skew
Market verdict & upside/downside classification
▼
  • verdict object { text: "Market assumes...", scores: [35%, 40%, 20%, 5%] }
  • skew enum UPSIDE | DOWNSIDE | BALANCED (computed from hypothesis net direction)
  • skew_score integer (0-100) Confidence in skew direction: 60 = moderately upside, 85 = strongly upside
  • skew_computation string Deterministic: upside_sum = sum(N1 & upside-N2), downside_sum = sum(N3, N4 & downside-N2), net = upside - downside
Section 7: Discriminators & Tripwires
Diagnostic metrics & future catalysts
▼

Discriminators (3-10): Which hypotheses do they differentiate? What's the current reading?

Metric Diagnosticity Discriminates Current
EBITDA margin HIGH N1 vs N3 32% ✓
Capex/revenue MEDIUM N2 vs N4 4.1% ⚠

Tripwires (2-10): Future catalysts that could trigger position changes

Tripwire: "Q2 EBITDA < A$500M" → triggers exit from N2 base case
Catalyst: "Q1 production guidance cut" → timing/event
Valence: negative (downside surprise)
Section 8: Technical & Gaps
Technical analysis & coverage gaps
▼
  • technical.regime enum Uptrend | Downtrend | Sideways
  • technical.ma50_ma200 string "50-day at A$12.50, 200-day at A$11.80, bullish crossover"
  • technical.volatility float Annualised: 24% = moderate
  • gapCoverage array (10 rows) One row per evidence domain: { domain: "...", coverage: "Full|Good|Limited" }
  • analyticalLimitations string What couldn't be assessed and why

3. LLM System Prompts — The Research Generation Instructions

Click to expand each prompt and see exactly what instructions are given to Gemini and Claude.

A: EVIDENCE CREATION Gemini 2.0 Flash — Coverage Initiation (~20 sec)
▼
You are a senior equity research analyst at Continuum Intelligence conducting an initial coverage assessment. For each of the 10 evidence domains below, produce a structured evidence card with: - "finding": 2-4 sentences with specific numbers, dates, and citations - "tension": 1-2 sentence counter-narrative or information gap - "source": data source description - "tags": array of hypothesis support/contradict tags - "table": null or structured data table if quantitative The 10 domains (MUST produce all 10): 1. Corporate Communications (epistemic class: motivated) 2. Regulatory Filings & Financial Statements (verified) 3. Broker Research (modelled) 4. Competitor Disclosures (verified) 5. Economic Data (measured) 6. Alternative Data (observed) 7. Academic Research (peer-reviewed) 8. Media & Social (unverified) 9. Leadership & Governance (observed) 10. Ownership & Capital Flows (measured) Rules: - Cite dates, numbers, and sources. Never fabricate data. - If data is unavailable for a domain, state the gap explicitly in "tension". - Australian English. No em-dashes. Use en-dashes or restructured sentences. - Each finding must reference specific data from the provided context. - Structure your response as JSON: { "evidence_cards": [ { "domain": "...", "finding": "...", "tension": "...", "source": "...", "tags": [...], "table": null } ] }
B: EVIDENCE UPDATE Gemini 2.0 Flash — Refresh Cycle (~20 sec)
▼
You are updating evidence cards for an existing equity research coverage. Given: - Existing 10 evidence domain cards (current state) - New data since last update (announcements, reports, news) - Current market price and volatility For each existing card, assess whether the new data warrants a material update. Rules: - Only update a card if new data is MATERIAL (price-moving or thesis-affecting). - Set "material_change": true only for genuine updates. - Preserve the existing finding/tension structure unless new data fundamentally changes it. - Cite dates, numbers, sources in any updates. - Use macro commodity prices (iron ore, copper, oil, gold) for resource/commodity companies. - Australian English. No em-dashes. - If no material change, return: { "domain": "...", "material_change": false } Structure your response as JSON: { "updated_cards": [ { "domain": "...", "material_change": true/false, "finding": "...", "tension": "...", "source": "...", "tags": [...] } ] }
C: HYPOTHESIS SYNTHESIS Claude Sonnet 4.6 — Coverage Initiation (~80 sec)
▼
You are a senior equity research analyst conducting initial ACH (Analysis of Competing Hypotheses) coverage. Given: - Company overview and sector context - Current market price and recent price history - 10 evidence domain cards (or data to assess them) Produce: 1. FOUR HYPOTHESES (N1-N4): - N1: Bull case (upside direction) - N2: Base case (usually highest weight, any direction) - N3: Bear case (downside direction) - N4: Tail risk (low probability extreme, any direction) Each hypothesis MUST have: - title: "N1: [descriptive 6-8 word title]" - direction: "upside" or "downside" or "neutral" - score: integer percentage as string (e.g., "35%") - description: 2-3 sentence scenario description - supporting: array of 2-4 supporting evidence lines - contradicting: array of 1-3 contradicting evidence lines - requires: what must be true for this scenario to play out CONSTRAINT: Scores MUST sum to 100%. 2. embedded_thesis (3-5 sentences): What does the current price assume? Reference the actual share price. 3. skew_description (2-3 sentences): Why is it upside/downside/balanced? 4. Narrative sections: - theNarrative: Dominant theme right now (3-5 sentences) - priceImplication: { summary: "...", details: ["bullet1", "bullet2", ...] } - evidenceCheck: Is the evidence base strong or weak? - narrativeStability: robust | fragile | uncertain 5. verdict: { text: "What the market is pricing and whether it's correct", scores: [35%, 40%, 20%, 5%] } 6. next_decision_point: { event: "Q2 earnings", date: "2026-07-15", metric: "EBITDA", thresholds: { positive: ">$5.2B", negative: "<$4.8B" } } 7. position_in_range: { bear: 10.50, base: 12.25, bull: 15.00, catalyst: 18.75 } (plain numbers in A$) 8. company_description: 4-6 sentence overview of what the company is and does Rules: - Reference the current price throughout. Update it to today's date. - All dates in next_decision_point must be future. - Be opinionated. Take a view on probability. Don't hedge with 25/25/25/25. - No emoji anywhere. - Australian English, no em-dashes. - Distinguish between N1 upside and N2 base; don't duplicate logic. Return as JSON: { "hypotheses": [ ... ], "embedded_thesis": "...", "skew_description": "...", "theNarrative": "...", "priceImplication": {...}, "evidenceCheck": "...", "narrativeStability": "...", "verdict": {...}, "next_decision_point": {...}, "position_in_range": {...}, "company_description": "..." }
D: HYPOTHESIS UPDATE Claude Sonnet 4.6 — Refresh Cycle (~80 sec)
▼
You are updating an existing ACH equity research coverage. Given: - Current hypothesis weights: [existing scores] - Updated evidence cards: [10 cards with material_change flags] - New market data: [current price, price history, announcements, news] - Current date and time context Reassess hypothesis probabilities in light of new evidence. Rules: - Scores MUST sum to 100% (integer percentages). - Only change scores if evidence warrants it. Don't move scores for the sake of it. - Explain score changes in a rationale line. - Rewrite ALL narrative sections to reflect current reality. - next_decision_point MUST be a future date (not past). - Reference the current market price, not stale prices from prior updates. - If evidence is ambiguous or contradictory, state the ambiguity explicitly. - Don't invent data. Use only what's provided. Update and return: { "hypotheses": [ ... ], "score_changes_rationale": "...", "embedded_thesis": "...", "skew_description": "...", "theNarrative": "...", "priceImplication": {...}, "verdict": {...}, "next_decision_point": {...}, "position_in_range": {...} }
E: STRUCTURE UPDATE Gemini 2.0 Flash — Refresh Cycle (~15 sec)
▼
Update the structural analysis sections (discriminators, tripwires, technical analysis, gaps) based on new market data. Return null for any section with no material change. For discriminators (3-10 rows): - diagnosticity: "HIGH" | "MEDIUM" | "LOW" - evidence: the specific data point or metric - discriminatesBetween: "N1 vs N3" format (which hypotheses does this differentiate?) - currentReading: current level with interpretation - scoreColor: "td-green" (bullish) | "td-amber" (neutral) | "td-red" (bearish) For tripwires (2-10 cards): - description: what event or metric triggers the wire - valence: "positive" | "negative" (upside or downside surprise?) - hypotheses_affected: which N1-N4 are affected - date_context: "Q2 2026" or "within 90 days" For technical_analysis: - regime: "Uptrend" | "Downtrend" | "Sideways" - ma50_ma200: current moving average relationship - volatility: annualised percentage - relative_performance: vs ASX200 (outperforming/underperforming/in-line) - commentary: 2-4 sentences For gaps: - gapCoverage: array of 10 rows, one per evidence domain - analyticalLimitations: what couldn't be assessed and why Return as JSON: { "discriminators": [ ... ], "tripwires": [ ... ], "technical_analysis": {...}, "gapCoverage": [ ... ], "analyticalLimitations": "..." }

4. Validation & Auto-Fix Rules

12 Auto-Fix Corrections (Applied Before Save)

  • String prices → float conversion
  • Gap percentage normalisation (0-100 scale)
  • Missing currentPrice filled from priceHistory[-1]
  • Empty scoreColor derived from hypothesis direction
  • Missing indexReturn defaults to 6.5%
  • relativeReturn recalculated (stockReturn - indexReturn)
  • Mojibake cleanup (recursive through all strings)
  • Emoji stripping (recursive through all strings)
  • heroCompanyDescription trim (max 600 chars)
  • Missing confidenceClass filled (high|medium|low)
  • Footer count correction (evidence card count)
  • Hypothesis score string normalisation ("%")

19 Validation Checks (Logged, Not Blocking)

  • Ticker uppercase
  • Company name non-empty
  • Hypotheses count = 3-4
  • Hypothesis scores sum to 100%
  • Evidence cards >= 5
  • Verdict text present & > 50 chars
  • Narrative text present & > 100 chars
  • Skew direction valid (UPSIDE|DOWNSIDE|BALANCED)
  • Position in range worlds = 4
  • Discriminators 3-10
  • Tripwires 2-10
  • Gap coverage rows = 10
  • Evidence intro >= 50 chars
  • No em-dashes (regex check)
  • All prices are numbers
  • Identity rows > 0
  • Hero metrics valid & present
  • Verdict score count matches hypothesis count
  • Verdict/hypothesis score divergence auto-fix

Skew Computation Logic:
1. Upside Sum = N1 (if upside direction) + any N2/N4 with upside direction
2. Downside Sum = N3 (if downside direction) + N4 (if downside direction) + any N2 with downside direction
3. Net = Upside Sum − Downside Sum
4. If Net > 5: UPSIDE | If Net < −5: DOWNSIDE | If −5 ≤ Net ≤ 5: BALANCED
5. Skew Score = Abs(Net) × 6 (capped at 100)
